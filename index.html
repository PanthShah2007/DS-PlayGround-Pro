Arrays & Dynamic Arrays (Vectors)

An array is a fixed-size, contiguous block of memory — every element sits right next to the last, so jumping to index i is a single pointer calculation: O(1).

A vector (C++ vector, Python list, JS Array) is a dynamic array. It over-allocates capacity, and when it runs out, it allocates a new block — usually double the size — copies every element over, and frees the old block.

That resize is an expensive O(n) operation, but it happens rarely enough (capacity doubling means it happens O(log n) times over n insertions) that the amortized cost of push_back stays O(1).

  Inserting or deleting at the front of an array/vector is still O(n) — every later element has to shift.

Strings

A string is, under the hood, an array of characters — so most array intuition carries over directly: indexing is O(1), and appending to the end is amortized O(1) if the string is backed by a growable buffer.

In C++, std::string is mutable and growable much like a vector. In Python and Java, strings are immutable — every "modification" (concatenation, replace, slicing) actually builds a brand-new string, which is why repeatedly concatenating in a loop is a classic O(n²) trap.

Hash Maps (Dictionaries)

A map stores key → value pairs. It runs the key through a hash function that turns it into an array index, so lookup skips straight to (roughly) the right bucket instead of scanning everything.

Collisions — two keys hashing to the same bucket — are handled by chaining (a small list per bucket) or open addressing (probe to the next free slot). A good hash function keeps buckets short, which is what keeps operations close to O(1) on average.

Worst case degrades to O(n) when many keys collide into the same bucket — rare with a decent hash function, but the reason languages randomize hash seeds for security.

Sorting Algorithms

Every sort trades off between time, space, and stability (whether equal elements keep their original relative order). Comparison-based sorts can't beat O(n log n) in the average case — that bound comes straight from the fact that there are n! possible orderings and each comparison only rules out half of them.

Bubble / Selection / Insertion

Simple O(n²) sorts. Insertion sort is genuinely fast on nearly-sorted data — many languages use it for small sub-arrays inside a bigger sort.

Merge Sort

Divide, sort each half recursively, merge. Guaranteed O(n log n) — the trade-off is an extra O(n) buffer for merging.

Quick Sort

Pick a pivot, partition around it, recurse. Fast in practice and in-place, but a bad pivot choice degrades to O(n²).
